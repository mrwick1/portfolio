The Case Against SPAs in 2026

I build single-page applications for a living. My own portfolio is plain HTML. That's not a contradiction.

Published Feb 25, 2026
Reading Time 8 min read
Tags
Architecture Performance Opinion
01The Confession

I write React professionally. I've migrated production apps to Next.js, wired up Redux stores, debugged hydration mismatches at 2 AM, and shipped server components before most teams had heard of them. React pays my rent. I like React.

And when it came time to build my own portfolio — the site you're reading right now — I opened a terminal and wrote <!DOCTYPE html>.

No npx create-next-app. No npm install. No package.json at all. This entire site is hand-written HTML, vanilla CSS, and about 40 lines of JavaScript for a theme toggle and scroll animations. There is no build step. There is no node_modules directory. I edit a file, push to Vercel, and it's live.

This isn't a contrarian hot take for engagement. It's the conclusion I reached after years of building SPAs and watching the industry reach for React the way a carpenter reaches for a hammer — reflexively, for every problem, regardless of whether the problem involves a nail.

02The Default Is Wrong

Somewhere in the last decade, SPAs became the default for everything. Portfolio? React. Blog? Next.js. Restaurant menu with three pages? Gatsby. A landing page that could be 4 KB of HTML became a 300 KB JavaScript bundle that renders a loading spinner before showing the same static content.

The problem isn't that SPAs exist. They solve real problems for real applications — complex state management, real-time updates, rich interactivity that would be painful to build with server round-trips. The problem is that we stopped asking whether the thing we're building actually has those problems.

The JavaScript Tax

Every SPA pays a tax before a single pixel appears on screen. The browser has to download the JavaScript bundle, parse it, compile it, and execute it before it can even start rendering your content. On a fast MacBook with fiber internet, this takes milliseconds. On a three-year-old Android phone on a 3G connection in Mumbai, it takes seconds. Real, noticeable, user-losing seconds.

This isn't theoretical. Google's research consistently shows that 53% of mobile users abandon pages that take longer than 3 seconds to load. When your marketing page or blog post ships 400 KB of JavaScript to render text and images, you're paying the JavaScript tax on content that has zero interactivity requirements.

HTML doesn't have this problem. The browser starts rendering HTML the moment it arrives. No parsing step. No compilation step. No hydration step. The content is the payload.

The Dependency Treadmill

Run npx create-next-app today and you'll get 300+ MB of node_modules before you've written a line of code. That's 300 MB of other people's code you now depend on, need to keep updated, and occasionally need to debug when things break in ways your code didn't cause.

I've watched teams spend entire sprints upgrading from Next.js 13 to 14 — not because the upgrade gave them anything their users needed, but because staying on the old version meant accumulating security vulnerabilities and falling behind on ecosystem compatibility. The framework becomes a second product you maintain alongside the one you're actually trying to build.

My portfolio has zero dependencies. It will work the same way in a browser today as it will in a browser five years from now, because HTML and CSS don't ship breaking changes.

The Complexity Trap

SPAs introduce categories of problems that don't exist in traditional websites. Client-side routing means you need to manage browser history, handle deep linking, and deal with the back button. State management means you need stores, reducers, or context layers. Code splitting means you need loading states for every route. Server-side rendering means you need to handle hydration mismatches, distinguish between server and client code, and debug issues that only appear in production.

These are all solvable problems. I solve them professionally every day. But if your website is a portfolio, a blog, a docs site, or a marketing page — you've introduced every one of these problems to display content that doesn't change between requests.

Instant Rendering

There is no JavaScript that needs to execute before you see content. The browser receives HTML and starts painting immediately. No loading spinner. No white flash. No layout shift as components hydrate. First Contentful Paint is limited only by network latency and the size of the HTML file — which, without a framework's runtime, is small.

Performance Without Trying

I didn't need to configure code splitting, tree shaking, lazy loading routes, or dynamic imports to get good performance. The CSS is in a <style> tag in the <head> so there's no render-blocking external request. The fonts are self-hosted with font-display: swap and preload hints. The only JavaScript is an IIFE at the bottom of the page for theme toggling and scroll-triggered fade-ins. Off-screen sections use content-visibility: auto so the browser skips rendering them until they're near the viewport.

These are the same performance techniques you'd use in a framework app, except here they required zero configuration, zero plugins, and zero build pipeline. I just wrote CSS.

Perfect SEO by Default

Every page is real HTML with real content in the source. No server-side rendering required because there's no client-side rendering to work around. Crawlers see exactly what users see. Structured data, Open Graph tags, canonical URLs, a sitemap — all of it just works because there's no JavaScript layer between the server and the content.

Durability

This is the one people underestimate. A React app I built in 2019 would need significant work to run today. CRA is deprecated. React Router has had multiple breaking API changes. The Webpack config would need updating. Half the dependencies would have security advisories.

An HTML file I wrote in 2019 would render identically in a 2026 browser. HTML is the most stable technology in the web stack. By building on it directly, I've built on the only layer that genuinely doesn't break.

No Build, No Deploy Pipeline Complexity

My deployment configuration is five lines of JSON:

{
  "buildCommand": "",
  "outputDirectory": ".",
  "headers": [{
    "source": "/assets/(.*)",
    "headers": [{ "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }]
  }]
}

No build command. The output directory is the repository root. Vercel serves the files as-is with a one-year cache on assets. There's nothing to optimize because there's nothing to build. The absence of a build step isn't a limitation — it's a feature. There's no CI that can fail, no build cache to invalidate, no mysterious production-only bugs caused by minification or tree shaking gone wrong.

04The Tradeoffs Are Real

I'm not going to pretend this approach has no downsides. It does, and you should know them before choosing it.

Repetition

My navigation, footer, theme toggle JavaScript, and CSS design system are duplicated across every HTML file. When I update the footer links, I update them in 10 places. There are no components, no partials, no template inheritance. This is the single biggest cost of going framework-free, and it's a real one.

For a 10-page portfolio, it's manageable. For a 50-page docs site, it would be painful. For a 500-page content site, it would be untenable. Know your scale before choosing your tools.

No Dynamic Features

There's no contact form that submits to an API. No comments section. No search. If I wanted any of these, I'd need to either add client-side JavaScript that talks to external services, or accept that some pages need a framework. The static approach works precisely because a portfolio doesn't need dynamic features.

Manual Everything

Adding a new case study means copying an HTML template and manually filling in the content. There's no CMS, no markdown-to-HTML pipeline, no generateStaticParams that builds pages from a data source. The site is artisanal in the most literal sense — each page is hand-crafted. This is fine at my scale. It would not be fine at a different scale.

05When SPAs Still Win

I'm not arguing that SPAs are bad. I'm arguing that they're a tool, not a default. Here's when they're the right tool:

  • Complex interactive applications. Dashboards with real-time data, collaborative editors, apps with drag-and-drop and undo/redo stacks. If the UI has state that changes frequently based on user actions, you need a reactive rendering model.
  • Apps that feel like apps. If users expect instant navigation, optimistic updates, offline support, and persistent UI state across views, an SPA delivers that experience in a way server-rendered pages can't match.
  • Large teams with shared component systems. When 20 developers need to build consistent UIs from shared primitives, a component framework provides structure, typing, and composition patterns that raw HTML doesn't offer.
  • Frequent content updates from dynamic sources. When pages are generated from APIs, databases, or CMS content that changes hourly, the build-and-serve model of static sites becomes impractical without static generation + revalidation (which, yes, is a framework feature).

I use React at work because the products I build have all of these characteristics. A SaaS dashboard with real-time collaboration, complex form flows, and 200+ shared components is exactly the use case React was designed for. The mistake isn't using React for that — it's using React for everything else too.

06The Real Question

Before you npm init your next project, ask one question: does this website have state?

Not "could it theoretically have state someday." Not "would it be cool if it had a dark mode toggle" (you can do that in 10 lines of vanilla JS). Does the core experience of this website require managing state that changes in response to user interaction, across multiple views, in real time?

If the answer is no — if you're building a portfolio, a blog, a landing page, a documentation site, a restaurant menu, a conference schedule, an event page, a personal homepage — you probably don't need a framework. You need HTML, CSS, and maybe a little JavaScript.

If the answer is yes — if you're building a dashboard, an editor, a chat app, an e-commerce platform with cart state and checkout flows — then yes, reach for React or Vue or Svelte or whatever your team knows. These tools exist because the problems they solve are real.

The problem with 2026 frontend development isn't the tools. The tools are extraordinary. The problem is that we've forgotten it's okay to not use them. We've conflated simplicity with amateurism, as if choosing HTML over React is choosing a bicycle over a car. But if you're going to the corner store, the bicycle is faster.

The best technology choice is the one that solves your actual problem with the least complexity. Sometimes that's a server component. Sometimes it's a <p> tag.
